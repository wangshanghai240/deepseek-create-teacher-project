一、Git 简介与核心概念

1.1 什么是 Git？

Git 是目前世界上最先进的 分布式版本控制系统（DVCS），由 Linux 之父 Linus Torvalds 于 2005 年创建。最初是为了更好地管理 Linux 内核源码而开发的。

1.2 为什么选择 Git？

  • 分布式：每个开发者本地都有完整的仓库副本，即使服务器离线也能正常工作
  • 分支轻量：创建和切换分支几乎是瞬时操作，鼓励频繁分支
  • 数据完整性：所有数据在存储前都经过 SHA-1 哈希校验，保证内容不被篡改
  • 暂存区机制：允许精细控制每次提交的内容
  • 强大的社区：GitHub、GitLab、Gitee 等平台广泛使用

1.3 Git 的三个区域

区域说明
工作区 (Working Directory)你电脑上看到的实际文件
暂存区 (Staging Area / Index)存放即将提交的文件快照，位于 .git/index
本地仓库 (Repository).git 目录，存储所有历史版本数据

1.4 Git 文件的四种状态

状态含义
未跟踪 (Untracked)文件未被 Git 管理，新增文件默认此状态
已跟踪 (Tracked)
  • 未修改 (Unmodified)：文件自上次提交后未改动
  • 已修改 (Modified)：文件已被修改但尚未暂存
  • 已暂存 (Staged)：修改已添加到暂存区，等待提交
工作区 ──git add──▶ 暂存区 ──git commit──▶ 本地仓库 ──git push──▶ 远程仓库
   ▲                    │
   └────git checkout────┘

2.2 初次配置

安装完 Git 后，第一件事就是设置用户名和邮箱——每次提交都会携带这些信息。

2.3 配置级别

级别作用域配置文件路径命令
--system系统所有用户/etc/gitconfig需管理员权限
--global当前用户所有仓库~/.gitconfig 或 ~/.config/git/config最常用

2.4 常用配置项

# 启用颜色输出

# 查看所有配置

# 查看某个配置项

三、基础操作：初始化、暂存与提交

3.1 创建新仓库

# 方式一：在当前目录初始化
cd my-project
git init

# 方式二：克隆远程仓库
git clone https://github.com/user/repo.git
git clone git@github.com:user/repo.git        # SSH 方式
git clone https://github.com/user/repo.git my-folder  # 指定目录名

3.2 查看状态

# 查看工作区和暂存区状态（最常用命令之一）
git status

# 简洁模式
git status -s    # 或 --short
# 输出示例：
#  M README.md    ← 已修改未暂存（空格 已暂存状态）
# M  index.html   ← 已修改已暂存（M 在第一位）
# ?? newfile.txt  ← 未跟踪文件

3.3 暂存文件（git add）

git add filename.txt        # 暂存单个文件
git add src/               # 暂存整个目录
git add .                  # 暂存当前目录所有变更（不含删除）
git add -A                 # 暂存所有变更（包括删除），等同于 git add --all
git add -p                 # 交互式逐块暂存（精细控制每次提交内容）

3.4 提交（git commit）

# 基本提交
git commit -m "feat: add login page"

# 跳过暂存区，直接提交已跟踪文件的修改（等同于 git add + git commit）
git commit -a -m "fix: resolve login bug"

# 修改最近一次提交的消息
git commit --amend -m "新提交信息"

# 向最近一次提交追加文件（不修改提交信息）
git add forgotten-file.js
git commit --amend --no-edit
💡 提交规范推荐： 使用 Conventional Commits 格式：
<type>: <简短描述>
type 包括：feat（新功能）、fix（修复）、docs（文档）、style（格式）、refactor（重构）、test（测试）、chore（杂项）

3.5 查看提交历史

git log                         # 完整历史
git log --oneline              # 简洁模式（一行一个提交）
git log --graph                # 图形化显示分支
git log --oneline --graph --all  # 黄金组合
git log -p                     # 显示每次提交的 diff
git log --author="tom"         # 按作者筛选
git log --since="2024-01-01"  # 按时间筛选
git log --grep="fix"          # 按提交信息搜索
git log -n 5                  # 只看最近 5 条
git reflog                    # 查看所有 HEAD 移动记录（救命的命令！）

四、分支管理

4.1 分支简介

分支是 Git 的核心特性。Git 的分支本质上只是一个指向某次提交的 可变指针。创建新分支的成本极低，因此在 Git 中鼓励频繁使用分支。

# 查看分支
git branch                     # 查看本地分支（* 表示当前分支）
git branch -r                 # 查看远程分支
git branch -a                 # 查看所有分支（本地+远程）
git branch -v                 # 查看分支及最新提交

4.2 创建与切换分支

# 创建分支
git branch feature-login

# 切换分支
git checkout feature-login

# 创建并切换（一步完成）
git checkout -b feature-login

# 新方式（推荐）
git switch feature-login        # 切换
git switch -c new-branch       # 创建并切换

4.3 删除分支

# 删除本地分支
git branch -d feature-login    # 安全删除（已合并）
git branch -D feature-login    # 强制删除（未合并也可删除）

# 删除远程分支
git push origin --delete feature-login
# 或
git push origin :feature-login

4.4 重命名分支

# 重命名当前分支
git branch -m new-name

# 重命名其他分支
git branch -m old-name new-name

五、远程仓库

5.1 管理远程仓库

# 查看远程仓库
git remote -v                  # 显示详细 URL

# 添加远程仓库
git remote add origin https://github.com/user/repo.git

# 修改远程仓库 URL
git remote set-url origin https://github.com/user/repo.git

# 删除远程仓库
git remote remove origin

5.2 推送（git push）

# 推送到远程（并建立跟踪关系）
git push -u origin main       # -u = --set-upstream

# 之后可以简写
git push

# 推送所有本地分支
git push --all origin

# 推送标签
git push origin --tags

# 强制推送（谨慎使用！）
git push --force origin main   # 会覆盖远程历史
git push --force-with-lease    # 更安全的强制推送（检查远程是否有新提交）
⚠️ 强制推送风险： 如果别人也在用同一个分支，--force 会删除他们的提交！始终优先使用 --force-with-lease。

5.3 拉取（git pull & git fetch）

# git fetch：只下载远程数据，不自动合并
git fetch origin
git fetch origin main         # 只获取 main 分支

# git pull = git fetch + git merge
git pull origin main

# git pull 用 rebase 代替 merge
git pull --rebase origin main
# 或者设置默认行为
git config --global pull.rebase true

六、撤销与回退

6.1 撤销工作区修改

# 撤销单个文件的修改（回到最近一次 git add 或 git commit 的状态）
git restore filename.txt
# 旧版语法
git checkout -- filename.txt

6.2 撤销暂存

# 将文件从暂存区移回工作区（保留修改）
git restore --staged filename.txt
# 旧版语法
git reset HEAD filename.txt

# 撤销所有暂存的文件
git restore --staged .

6.3 撤销提交

# 方式一：git reset —— 回退到指定提交（会删除之后的提交记录）
git reset --soft HEAD~1       # 回退1个提交，修改留在暂存区
git reset --mixed HEAD~1       # 回退1个提交，修改留在工作区（默认）
git reset --hard HEAD~1       # 回退1个提交，彻底丢弃修改（⚠️ 不可恢复！）

# 方式二：git revert —— 创建新的反提交（安全，不修改历史）
git revert HEAD                # 撤销最近一次提交
git revert HEAD~3             # 撤销倒数第4次提交
💡 reset vs revert：
git reset 会修改提交历史，适用于本地尚未推送的提交。
git revert 会创建新的提交来抵消旧提交，不会修改历史，适用于已推送的提交。

6.4 恢复已删除的分支或提交

# 查看所有历史操作（包括已删除的分支和已 reset 的提交）
git reflog

# 从 reflog 中恢复
git checkout -b recovered-branch HEAD@{2}
# 或
git reset --hard HEAD@{2}

七、合并与变基

7.1 合并（git merge）

# 将 feature 分支合并到当前分支
git checkout main
git merge feature

# 创建合并提交（即使可以快进）
git merge --no-ff feature

# 取消合并（有冲突时）
git merge --abort

7.2 变基（git rebase）

# 将当前分支的提交"移植"到目标分支的顶端
git checkout feature
git rebase main

# 交互式变基（修改历史记录）
git rebase -i HEAD~3          # 修改最近3条提交
# 交互式变基支持的操作：
# pick    - 保留提交
# reword  - 修改提交信息
# edit    - 修改提交内容
# squash  - 合并到上一个提交
# fixup   - 合并到上一个提交（丢弃信息）
# drop    - 删除提交

7.3 解决冲突

当合并或变基出现冲突时，Git 会在冲突文件中标记冲突区域：

<<<<<<< HEAD
这是当前分支的内容
=======
这是合并进来的分支的内容
>>>>>>> feature

解决步骤：

  1. 打开冲突文件，手动编辑保留想要的内容
  2. 删除冲突标记 <<<<<<<、=======、>>>>>>>
  3. 保存文件后执行 git add 标记为已解决
  4. 执行 git commit（合并）或 git rebase --continue（变基）
# 变基冲突时的其他操作
git rebase --continue   # 冲突解决后继续
git rebase --skip       # 跳过当前提交
git rebase --abort      # 取消变基，回到之前状态
⚡ Git Flow 黄金法则： 不要对已推送的公共分支（如 main、develop）执行 rebase！它会导致提交历史重写，影响所有协作者。 
# Cherry-pick：挑选另一个分支的特定提交到当前分支
git cherry-pick commit-hash
git cherry-pick hash1 hash2   # 挑选多个
git cherry-pick A..B           # 挑选 A 之后到 B 之间的所有提交

八、标签管理

8.1 创建标签

# 轻量标签（仅一个引用）
git tag v1.0.0

# 附注标签（包含作者、日期、信息，推荐）
git tag -a v1.0.0 -m "Release version 1.0.0"

# 为历史提交打标签
git tag -a v0.5.0 commit-hash -m "Beta release"

8.2 查看与推送标签

git tag                        # 列出所有标签
git tag -l "v1.*"             # 按模式搜索标签
git show v1.0.0               # 查看标签详情

# 推送标签到远程（标签不会随 git push 自动推送）
git push origin v1.0.0        # 推送单个标签
git push origin --tags        # 推送所有标签

# 删除标签
git tag -d v1.0.0             # 删除本地标签
git push origin --delete v1.0.0  # 删除远程标签

九、.gitignore 文件

9.1 什么是 .gitignore？

.gitignore 文件告诉 Git 哪些文件不需要纳入版本管理。通常用于排除：编译产物、依赖文件夹、环境配置、系统文件等。

9.2 常见规则

# 注释

# 忽略 node_modules 文件夹
node_modules/

# 忽略所有 .log 文件
*.log

# 但不要忽略 important.log
!important.log

# 忽略 build 目录下所有内容
build/

# 忽略当前目录下的 config.env（不含子目录）
/config.env

# 忽略任何地方的 .env 文件
.env

# 忽略 .DS_Store（macOS 系统文件）
.DS_Store

# 忽略 Thumbs.db（Windows 缩略图缓存）
Thumbs.db

# 忽略 dist 文件夹
dist/

9.3 Git 忽略规则优先级

  1. .gitignore 文件（项目级别，提交到仓库中）
  2. .git/info/exclude（项目级别，不提交）
  3. ~/.gitignore_global（全局级别，需配置 git config --global core.excludesfile）
💡 注意： 如果文件已经被 Git 跟踪，后续添加 .gitignore 规则不会生效。需要先执行 git rm --cached <file> 停止跟踪。

十、工作流模型

10.1 GitHub Flow（最简单，适合持续部署）

main ──●────●────●────●────●
        \        /    \        /
         ●──●──●      ●──●──●
        feature1      feature2
  1. main 分支始终保持可部署状态
  2. 从 main 创建功能分支：git checkout -b feature-x
  3. 在功能分支上开发并提交
  4. 发起 Pull Request（PR）进行代码审查
  5. 通过后合并到 main，立即部署

10.2 Git Flow（较复杂，适合版本发布）

develop ──●─────────●─────────●─────────●
            \       / \       / \       /
feature ──●──●──●    ●──●──●   ●──●──●
                          \
release ──────────────●───●
                          \
hotfix ───────────●─────────●
                  \         /
main ─────────────●───────────●─────────●
  • main — 生产就绪版本，只接受合并
  • develop — 日常开发主分支
  • feature/* — 功能开发分支，合并到 develop
  • release/* — 发布准备分支，只修 bug
  • hotfix/* — 紧急修复分支，从 main 创建，修复后合并到 main 和 develop

10.3 GitLab Flow

结合 GitHub Flow 和 Git Flow，增加了环境分支（如 pre-production、production），适合有多个环境的部署流程。

十一、常用高级技巧

11.1 暂存工作（git stash）

# 暂存当前工作区修改（切换到其他分支前使用）
git stash

# 暂存并包含未跟踪文件
git stash -u                   # 或 --include-untracked

# 查看暂存列表
git stash list

# 恢复最近一次暂存（并删除）
git stash pop

# 恢复最近一次暂存（不删除）
git stash apply

# 恢复指定的暂存
git stash apply stash@{2}

# 删除指定暂存
git stash drop stash@{2}

# 清除所有暂存
git stash clear

11.2 比较差异（git diff）

git diff                        # 工作区 vs 暂存区
git diff --staged              # 暂存区 vs 最近提交
git diff HEAD                  # 工作区 vs 最近提交
git diff main feature         # 比较两个分支
git diff commit1 commit2      # 比较两个提交
git diff --stat                # 仅显示变更文件列表
git diff --name-only           # 仅显示文件名

11.3 二分查找（git bisect）

# 用二分法查找引入 bug 的提交
git bisect start
git bisect bad                 # 当前提交有问题
git bisect good v1.0           # v1.0 是好的
# Git 会 checkout 一个中间提交，测试后标记 good/bad
git bisect good
git bisect bad
# 重复直到找到第一个坏提交
git bisect reset               # 结束 bisect

11.4 子模块

# 添加子模块
git submodule add https://github.com/user/lib.git lib

# 克隆包含子模块的项目
git clone --recursive https://github.com/user/project.git

# 更新所有子模块
git submodule update --init --recursive

11.5 清理文件

git clean -n                    # 预览将要删除的未跟踪文件
git clean -f                    # 删除未跟踪文件
git clean -fd                   # 删除未跟踪文件和目录

11.6 别名配置

git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.st status
git config --global alias.lg "log --oneline --graph --all"

# 使用别名
git lg       # 等价于 git log --oneline --graph --all
git ci -m "message"  # 等价于 git commit -m

11.7 归档导出

# 导出指定版本的文件（不带 .git 目录）
git archive -o project-v1.0.zip HEAD
git archive -o project-v1.0.tar.gz v1.0

十二、常见问题与解决方案

12.1 提交了敏感信息怎么办？

# 1. 从 Git 历史中彻底删除文件
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch config/secret.json" \
  --prune-empty --tag-name-filter cat -- --all

# 2. 或者使用更方便的工具（需要安装）
git install git-filter-repo
git filter-repo --path config/secret.json --invert-paths

# 3. 立即更换所有密码和密钥！
# 4. 强制推送覆盖远程历史
git push --force --all origin

12.2 撤销 git add 但不想丢失修改

git restore --staged filename.txt
# 文件修改会保留在工作区

12.3 撤销最近一次提交但保留修改

git reset --soft HEAD~1
# 修改会留在暂存区，可以重新提交

12.4 想修改历史中的某个提交信息

git rebase -i HEAD~5    # 交互式变基，将 pick 改成 reword

12.5 将多个提交合并成一个

git rebase -i HEAD~5    # 将 pick 改成 squash 或 fixup

12.6 git pull 时遇到冲突

# 方式一：解决冲突后提交
# 解决完后:
git add .
git commit

# 方式二：放弃合并，回到之前状态
git merge --abort

12.7 不小心删除了未推送的分支

git reflog   # 找到分支最后一次提交的 hash
git checkout -b recovered-branch commit-hash

12.8 远程仓库连接问题

# HTTPS 连接失败时切换为 SSH
git remote set-url origin git@github.com:user/repo.git

# SSH 连接失败时切换为 HTTPS
git remote set-url origin https://github.com/user/repo.git

# 配置代理（如果使用科学上网）
git config --global http.proxy http://127.0.0.1:7890
git config --global https.proxy http://127.0.0.1:7890

# 取消代理
git config --global --unset http.proxy
git config --global --unset https.proxy
